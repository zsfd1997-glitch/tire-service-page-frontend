## 当前已知业务边界

| 类别 | 结论 |
|---|---|
| 用户范围 | 先做单用户自助查看与操作，不展开后台管理端 |
| 页面范围 | 登录页、首页、车辆页、记录页、知识页、商品页、预约页 |
| 核心闭环 | 登录 -> 首页总览 -> 车辆/记录/知识/商品 -> 预约 |
| 非目标 | 支付、订单履约、后台管理端、品牌视觉自由探索 |
| 真相源优先级 | 现役后端接口 > PRD/TRD > 数据治理文档 > mock 样本 |

## 当前已知技术边界

| 类别 | 结论 |
|---|---|
| 接口返回 | 普通接口统一 `ApiResponse<T>`；聚合接口统一 `FrontendShell<T>` |
| 模块控制 | 首页和购买详情必须消费 `modules[moduleKey].state`，处理 `ready/hidden/conflict/unavailable` |
| 鉴权 | 登录后统一携带 `Authorization: Bearer <token>` |
| 数据口径 | 统一使用现役字段名：`userId/vehicleId/storeId/sourcePage/createdAt/updatedAt/spec/matchedSpecCode` |
| 规则配置 | `rule_configs` 已收紧为 `ruleKey/valueJson/valueType/unit/description` |
| 服务卡 | `HEALTHY` 默认隐藏；`REPLACE/ROTATE` 才可能显示提醒 |
| 预约 mode | 当前只允许 `PICKUP`、`SELF` |
| 默认登录 | 登录页默认填充演示账号 `admin / 123456`，优先支持一键进入系统 |

## 绝对不要做的事

| 禁止项 | 原因 |
|---|---|
| 发明平行字段 | 会把前后端 contract 再次拉散 |
| 忽略 `modules` | 会把隐藏或冲突模块错误渲染出来 |
| 把 `rule_configs.unit` 重新塞回 `description` | 会破坏我们刚完成的数据库治理 |
| 直接按 `knowledge_items` 猜首页知识逻辑 | 首页知识有配置层，不是内容表直出 |
| 把 AI 结果当成健康计算真相源 | 现役代码里 AI 只做辅助，不直接决定健康态 |
| 自作主张扩后台或支付链路 | 超出 MVP 范围 |

## 外部工具写前端前必须确认的未知项

| 项 | 当前状态 | 处理规则 |
|---|---|---|
| 前端技术栈 | 未在现役仓库中锁定 | 如果用户没明确指定，外部工具必须先声明它采用什么栈，再开始写代码 |
| 前端目录落点 | 仓库内还没有现役前端目录 | 不允许外部工具擅自大规模重构仓库；必须先说明建议落点 |
| 视觉目标图 | 当前没有 Figma 或高保真截图 | 默认按 PRD/TRD 做功能优先版，不要自由发挥品牌设计 |
| 运行方式 | 后端现役端口是 `3012`，前端运行方式未锁定 | 外部工具必须明确本地联调方式 |

## 外发任务的标准写法

| 模块 | 要求 |
|---|---|
| 任务目标 | 明确只写哪一页或哪几个组件 |
| 文件范围 | 指明允许新增/修改哪些目录和文件 |
| 依赖文档 | 至少附本文件 + 页面子包 + 验收包 |
| 输出要求 | 必须给出真实代码、文件清单、关键说明、浏览器验证方式 |
| 假设管理 | 缺信息时要先列 `known / unknown / assumption`，不能闷头乱猜 |

## 标准任务正文

```md
你现在要为 `tirebot / 轮胎助手` 项目编写用户端前端代码。

先阅读：
- 文档/治理/前端/前端包.md
- 文档/治理/前端/页面子包.md
- 文档/治理/前端/验收包.md
- 文档/prd/用户端PRD.md
- 文档/prd/用户端TRD.md

真相源：
- 业务目标和页面范围以 `用户端PRD.md` 为准
- 接口、组件 contract、状态机以 `用户端TRD.md` 为准
- 数据治理和字段约束以 `文档/治理/数据/治数原则.md`、`文档/治理/数据/测试矩阵.md` 为准
- 后端接口与 DTO 以 `Java后端/` 为准

你必须遵守：
- 不得发明平行字段
- 必须消费 `FrontendShell.modules`
- 必须处理 loading / empty / error / hidden / conflict / unavailable
- 登录页默认填充 `admin / 123456`
- 不得把 `rule_configs.unit` 塞回 `description`
- 不得扩展后台管理端、支付、订单履约

如果前端技术栈、目录落点、视觉目标图没有明确，请先列出：
- known
- unknown
- assumption

然后只实现这次指定的页面或组件：
- 【在这里替换成具体目标】

输出要求：
- 直接输出可落地代码
- 明确新增/修改哪些文件
- 解释每个关键改动
- 提供浏览器验证方式
```

## 使用顺序

| 步骤 | 动作 |
|---|---|
| 1 | 先发本文件，让对方建立项目级边界 |
| 2 | 再发 [`页面子包.md`](./页面子包.md)，限定它这次到底写哪些页面 |
| 3 | 最后发 [`验收包.md`](./验收包.md)，限定它做到什么算完成 |
| 4 | 对方回代码后，优先检查字段名、接口路径、模块状态、文件落点，再看 UI 细节 |
