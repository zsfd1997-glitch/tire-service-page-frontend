# 轮胎助手前端 PRD v1 完整稿

## 1. 文档定位
- 文档类型: 基于当前前端实现反推整理的产品需求文档（Frontend PRD）
- 文档目标: 把当前可运行前端抽象为可继续设计、重构、联调、拆任务的正式需求基线
- 文档读者:
  - 前端工程师
  - 设计师
  - 产品经理
  - AI coding agent
  - 后端联调同学
- 文档范围: 当前仓库内的用户端前端页面、路由、页面状态、交互行为、数据口径、组件职责
- 非目标:
  - 后台管理端
  - 支付与订单履约
  - 品牌营销官网
  - 实时后端实现细节

## 2. 产品一句话定义
`轮胎助手` 是一个面向车主的移动优先前端应用，用最短路径完成“登录 -> 查看车辆轮胎状态 -> 查看商品 -> 提交预约”的轮胎服务闭环。

## 3. 产品目标
### 3.1 业务目标
- 帮助用户快速理解当前车辆轮胎状态
- 在状态异常或用户有购买意图时，给出推荐商品
- 将用户引导到预约动作，形成服务转化

### 3.2 前端目标
- 用一套轻量、移动端友好的页面承载轮胎服务流程
- 让页面在没有真实后端时也能通过 mock 完整演示主要链路
- 保持字段名、页面关系、状态逻辑与后端契约一致

### 3.3 用户目标
- 不需要理解复杂轮胎知识也能知道“现在要不要处理”
- 能看到自己车辆对应的规格与推荐商品
- 能从首页直接进入预约

## 4. 用户与使用场景
### 4.1 目标用户
- 已绑定车辆的普通车主
- 希望通过手机完成轮胎信息查看和预约
- 对轮胎专业术语不熟，但愿意接受明确建议

### 4.2 使用场景
- 场景 A: 用户首次登录演示系统，快速了解产品能力
- 场景 B: 用户在首页切换车辆，查看不同车辆的轮胎状态
- 场景 C: 用户从异常状态进入商品或预约链路
- 场景 D: 用户维护轮胎更换记录
- 场景 E: 用户查看知识内容或让 AI 辅助识别磨损

## 5. 当前版本页面范围
当前实现实际包含以下页面与子页面:
- 登录页 `/login`
- 首页 `/home`
- 车辆详情页 `/vehicle/:id`
- AI 识别页 `/ai-inspection/:id`
- 轮胎规格选择页 `/tire-spec/:id`
- 商品列表页 `/products`
- 商品详情页 `/products/:id`
- 预约页 `/appointment`
- 记录页 `/records`
- 知识页 `/knowledge`
- 门店选择页 `/store-location`

## 6. 路由与页面关系
### 6.1 主路由
| 路由 | 页面 | 说明 |
|---|---|---|
| `/` | 重定向到首页 | 默认进入 `/home` |
| `/login` | 登录页 | 演示账号登录入口 |
| `/home` | 首页 | 用户总览页 |
| `/vehicle/:id` | 车辆详情页 | 单车状态与动作页 |
| `/ai-inspection/:id` | AI 识别页 | 上传轮胎图片做磨损识别 |
| `/tire-spec/:id` | 规格选择页 | 查看和选择轮胎规格 |
| `/products` | 商品列表页 | 按规格浏览商品 |
| `/products/:id` | 商品详情页 | 查看单个商品详情并去预约 |
| `/appointment` | 服务预约页 | 选择到店模式并提交 |
| `/records` | 换胎记录页 | 添加、查看、删除记录 |
| `/knowledge` | 知识页 | 分类浏览轮胎知识 |
| `/store-location` | 门店选择页 | 搜索/选择门店 |

### 6.2 别名与重定向
- `/tire-shop` -> `/products`
- `/tire/:id` -> `/products`
- `/service-booking` -> `/appointment`
- `/tire-record` -> `/records`
- `/faq` -> `/knowledge`
- `/tire-health` -> `/home`

## 7. 全局交互规则
### 7.1 页面滚动
- 路由切换后统一滚动到页面顶部
- 由 `RootLayout` 在 `pathname` 变化时触发

### 7.2 导航模型
- 页面分为两类:
  - 顶部 Header + 底部导航页
  - 独立流程页（登录）
- 底部导航固定 5 个入口:
  - 首页
  - 知识
  - 预约
  - 商城
  - 记录

### 7.3 视觉风格
- 移动端优先
- 卡片式布局为主
- 主色偏蓝
- 健康/警告/危险使用绿黄红区分

## 9. 页面级需求

### 9.1 登录页
#### 页面目标
- 让用户通过默认演示账号快速进入系统

#### 当前实现
- 默认账号: `admin`
- 默认密码: `123456`
- 支持显示/隐藏密码
- 支持回车登录
- 登录成功写入 `localStorage`
- 登录失败显示错误提示

#### 核心需求
- 用户看到品牌标题与轮胎视觉图
- 用户能直接点击“一键登录”
- 用户也可手动修改账号和密码
- 账号为空时拦截并提示“请输入账号”
- 密码为空时拦截并提示“请输入密码”
- 错误账号密码提示“账号或密码错误，请重试”

### 9.2 首页
#### 页面目标
- 展示当前车辆状态总览与下一步动作

#### 页面模块
- 顶部品牌区
- 车辆切换卡片
- 轮胎健康度卡片
- 每周知识卡片
- 轮胎小管家入口
- AI 磨损识别入口
- 商品推荐卡片
- 底部导航

#### 核心行为
- 可左右切换车辆
- 可通过圆点直接切换车辆
- 当前车辆变化后，健康信息、商品推荐、知识区均随车变化
- 健康卡可跳到规格选择页
- 健康卡可跳到记录页
- 健康卡可跳到预约页
- 商品卡点击进入详情页
- “一键购买”跳商品列表页

### 9.3 车辆详情页
#### 页面目标
- 展示单车详细状态与服务建议

#### 页面模块
- 标签切换
- 车辆基础信息卡
- 服务卡提醒
- 健康度分析区
- AI 识别入口
- 更换建议商品区

#### 核心行为
- 标签在“轮胎健康度 / 轮胎更换”间切换
- `HEALTHY` 且模块非 ready 时，不显示服务卡
- `REPLACE/ROTATE` 且模块 `ready` 时显示服务卡
- 用户点击“稍后处理”后，服务卡在当前会话隐藏

### 9.4 AI 识别页
#### 页面目标
- 让用户上传轮胎图片并查看识别结果

#### 当前实现
- 先上传图片
- 再点击识别
- 识别为模拟延时
- 根据车辆健康状态映射识别结果

### 9.5 规格选择页
#### 页面目标
- 展示并选择轮胎规格

#### 当前实现
- 从当前车辆规格解析出宽度、扁平比、轮毂尺寸
- 打开底部选择器后滚轮选择三列数值
- 最终拼接成完整规格

### 9.6 商品列表页
#### 页面目标
- 浏览商品并按规格筛选

#### 当前实现
- 顶部显示当前车辆及适配规格
- 可按规格筛选
- 商品卡支持收藏态切换
- 商品卡支持跳转详情

### 9.7 商品详情页
#### 页面目标
- 展示商品详细信息，并引导去预约安装

#### 当前实现
- 顶部 Hero 区展示商品视觉和促销
- 展示规格、适配性、价格、产品描述、正品保障、用户评价
- 底部固定 CTA 为“立即预约安装”

### 9.8 预约页
#### 页面目标
- 用户选择服务方式并提交预约

#### 当前实现
- 模式仅两种: `PICKUP` / `SELF`
- 提交后出现预约成功页
- 成功页展示车辆与服务方式摘要

### 9.9 记录页
#### 页面目标
- 添加与查看换胎记录

#### 当前实现
- 选择换胎时间
- 选择换胎里程
- 选择换胎数量
- 点击底部按钮新增记录
- 可删除单条记录

### 9.10 知识页
#### 页面目标
- 以分类 + 展开详情的形式查看知识内容

#### 当前实现
- 5 个分类标签
- 标签切换过滤列表
- 条目支持展开/收起
- 底部带服务承诺模块

### 9.11 门店选择页
#### 页面目标
- 搜索与选择服务门店

#### 当前实现
- 支持关键字搜索
- 支持单选门店
- 支持确认并返回上一页

## 10. 全局状态要求
- 登录页至少处理:
  - 默认值
  - 输入校验
  - loading
  - success
  - error
- 首页至少处理:
  - 切车
  - 商品推荐
  - 模块显隐
- 车辆页至少处理:
  - `HEALTHY`
  - `ROTATE`
  - `REPLACE`
  - 服务卡 ack
- 商品页至少处理:
  - 规格筛选
  - 适配商品
  - 空结果
- 预约页至少处理:
  - idle
  - loading
  - success
